# Algebra Solver — AI That Learns to Solve Equations

## Goal
Build a neural network that **learns to solve single-variable linear equations** by seeing thousands of examples — not by being programmed with math rules, but by discovering the patterns itself.

## What We're Building
A **sequence-to-sequence (seq2seq) model** — a neural network that takes a math equation as input and produces the solution as output, one character at a time.

```
Input:  "2x + 3 = 7"     →     Output: "x = 2"
Input:  "5x - 10 = 15"   →     Output: "x = 5"
Input:  "x / 4 = 3"      →     Output: "x = 12"
```

Think of it like this: instead of coding "subtract 3 from both sides, then divide by 2", we show the model **50,000 solved equations** and let it figure out the pattern.

## Why This Approach
- **Natural next step** from the text generator (same PyTorch, but new architecture)
- **Synthetic data** — we generate unlimited training examples with code (no manual data collection)
- **Verifiable results** — unlike text generation, math answers are objectively right or wrong
- **Same core idea as real AI** — models like GPT solve math by pattern recognition, not by "understanding" algebra
- Shows **encoder-decoder architecture** (the backbone of translation models, chatbots, etc.)

## How It Differs from SimpleAI Text Generator

| Aspect | Text Generator (SimpleAI) | Algebra Solver |
|---|---|---|
| Task | Continue text from a seed | Solve equation → answer |
| Architecture | Single RNN | Encoder RNN + Decoder RNN |
| Data source | Hand-written quotes (99 lines) | Programmatically generated (50,000 equations) |
| Vocabulary | 48 characters (letters, punctuation) | ~25 characters (digits, x, +, -, *, /, =, space) |
| Output type | Open-ended (any text) | Structured (always "x = number") |
| Evaluation | Subjective (does it "look good"?) | Objective (right or wrong answer) |
| New concepts | — | Encoder-decoder, teacher forcing, synthetic data, accuracy metrics |

## Requirements

### Functional
1. **Data Generation**: Programmatically generate 50,000+ linear equations with solutions
2. **Training**: Train an encoder-decoder model on equation→solution pairs
3. **Solving**: Given a new equation, predict the correct solution
4. **Evaluation**: Measure exact-match accuracy (% of equations solved correctly)
5. **Interactive**: A prompt where you type equations and see the model solve them

## Key New AI Concepts You'll Learn

| Concept | What It Means (Simply) |
|---|---|
| **Encoder-Decoder** | Two neural networks working together: one reads the input, the other writes the output |
| **Seq2Seq** | "Sequence to sequence" — turning one sequence of characters into another |
| **Teacher Forcing** | During training, feeding the correct answer to the decoder (instead of its own predictions) to speed up learning |
| **Synthetic Data** | Training data generated by code, not collected from the real world |
| **Special Tokens** | `<SOS>` (start of sequence) and `<EOS>` (end of sequence) — signals that tell the decoder when to start and stop |
| **Attention** (optional) | Letting the decoder "look back" at specific parts of the input while generating output |
| **Exact-Match Accuracy** | The % of test equations where the model's answer is exactly correct |
| **Train/Test Split** | Splitting data into training set (model learns from) and test set (model is evaluated on) |
| **Greedy Decoding** | Picking the most likely character at each step during generation |

### Step 05 — Encoder Model
**Branch:** `algebra-step-05-encoder`

**What:** Build the **encoder** — an RNN that reads the equation and compresses it into a fixed-size "context vector."

**Files:** `algebra/src/encoder.py`

**Architecture:**
```
"2x + 3 = 7"
     │
     ▼
┌──────────────────────────────────────────────────┐
│  Embedding(~23, 64)                              │
│  Each character → 64-dim vector                  │
│                                                  │
│  GRU(64, 128, 2 layers, bidirectional=True)      │
│  Reads equation left-to-right AND right-to-left  │
│  Output: 256-dim context per position            │
│                                                  │
│  Final hidden state = "context vector"           │
│  256-dim summary of the entire equation          │
└──────────────────────────────────────────────────┘
     │
     ▼
  context vector (256-dim)
  "This equation means: 2 times x, plus 3, equals 7"
```

**Why GRU instead of RNN?**
- GRU (Gated Recurrent Unit) is better at remembering important information
- It has "gates" that control what to remember and what to forget
- Works much better for math where every character matters

**Why bidirectional?**
- A regular RNN reads only left→right
- Bidirectional reads left→right AND right→left
- For "2x + 3 = 7", the "= 7" part at the end is crucial for understanding "2x + 3"
- Bidirectional lets the encoder see both directions

**What you'll learn:** Encoder architecture, GRU vs simple RNN, bidirectional processing.

**Validation:**
- Forward pass produces correct output shapes
- Hidden state has expected dimensions
- Bidirectional output is 2× the hidden size
- Model parameter count matches expectations

### Step 06 — Decoder Model
**Branch:** `algebra-step-06-decoder`

**What:** Build the **decoder** — an RNN that takes the context vector and generates the solution one character at a time.

**Files:** `algebra/src/decoder.py`

**Architecture:**
```
  context vector (from encoder)
     │
     ▼
┌──────────────────────────────────────────────────┐
│  Receives: <SOS> token + context vector          │
│                                                  │
│  Embedding(~23, 64)                              │
│  GRU(64, 256, 2 layers)                          │
│  Linear(256, ~23)                                │
│                                                  │
│  Step 1: <SOS> + context → predicts "x"          │
│  Step 2: "x"   + hidden  → predicts " "          │
│  Step 3: " "   + hidden  → predicts "="          │
│  Step 4: "="   + hidden  → predicts " "          │
│  Step 5: " "   + hidden  → predicts "2"          │
│  Step 6: "2"   + hidden  → predicts <EOS>        │
│                                                  │
│  Output: "x = 2"                                 │
└──────────────────────────────────────────────────┘
```

**Teacher forcing explained:**
- During **training**: we feed the correct next character (from the answer) to the decoder at each step, even if it predicted wrong — this speeds up learning dramatically
- During **inference**: we feed the decoder's own prediction back in (no cheating!)

```
Training (teacher forcing):
  Step 1: input=<SOS>  target="x"    model predicts "x" ✓ (or "3" ✗, doesn't matter)
  Step 2: input="x"    target=" "    ← always feed correct "x", regardless of prediction
  Step 3: input=" "    target="="

Inference (autoregressive):
  Step 1: input=<SOS>  → model predicts "x"   → feed "x" to next step
  Step 2: input="x"    → model predicts " "   → feed " " to next step
  Step 3: input=" "    → model predicts "="   → feed "=" to next step
```

**What you'll learn:** Decoder architecture, teacher forcing, autoregressive generation, `<SOS>`/`<EOS>` token usage.

**Validation:**
- Decoder accepts encoder's hidden state
- Forward pass produces correct output shapes
- Teacher forcing mode works (accepts target sequence)
- Generation mode works (produces output from `<SOS>`)
- `<EOS>` token stops generation

### Step 08 — Training Loop
**Branch:** `algebra-step-08-training`

**What:** Train the seq2seq model on 40,000 equations.

**Files:** `algebra/src/train.py`

**Training configuration:**
- Epochs: 30-50
- Batch size: 64
- Learning rate: 0.001
- Optimizer: Adam
- Loss: CrossEntropyLoss (ignoring `<PAD>` tokens)
- Teacher forcing ratio: starts at 1.0, decreases to 0.5 over training

**Key concept — teacher forcing schedule:**
```
Epoch  1-10:  teacher_forcing_ratio = 1.0   (always use correct answer)
Epoch 11-20:  teacher_forcing_ratio = 0.75  (75% correct, 25% own prediction)
Epoch 21-30:  teacher_forcing_ratio = 0.5   (50/50 mix)
```
This gradual decrease helps the model transition from "learning with help" to "solving on its own."

**What you'll learn:** Training a seq2seq model, teacher forcing schedules, ignoring padding in loss.

**Validation:**
- Loss decreases consistently over epochs
- Training completes under 10 minutes on CPU
- Model and vocabulary are saved to disk
- Loss history is saved for plotting

### Correctness Verification Method
Unlike the text generator (where "good output" is subjective), we can **mathematically verify** every answer:

```

Model says: x = 2
Verify: 2(2) + 3 = 4 + 3 = 7 ✓ (left side equals right side)
```

This substitution check runs automatically during evaluation and can be toggled in interactive mode.
